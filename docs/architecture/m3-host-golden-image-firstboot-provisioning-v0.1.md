# M3 主机黄金镜像、首启个性化与自动初始化开发策略 V0.1

- **状态**：已确认开发策略
- **适用对象**：温室环境监测系统 Home Assistant 主机（魔百盒/斐讯 T1 类 ARM64 主机）
- **部署基线**：Debian 13 + Docker Engine + Home Assistant Container
- **版本**：V0.1
- **日期**：2026-07-16

## 1. 背景与目标

项目计划在完成主机运行环境、基础服务和通用配置后制作整机黄金镜像，量产时批量复制，以最大限度减少用户安装、配置和维护负担。

本阶段不采用“把一台已经使用过的主机原样复制给所有用户”的方式，而采用：

> **黄金公共镜像 + 每台设备首次启动个性化 + 用户首次启用初始化**

目标是把普通用户可见操作控制在最低限度：

1. 将主机通过网线接入路由器；
2. 接通电源；
3. 手机 App 或浏览器发现 Home Assistant；
4. 创建用户自己的 Home Assistant 管理员账号；
5. 系统自动完成 MQTT、manager、节点发现和仪表盘初始化。

用户不需要理解 Linux、Docker、MQTT、Home Assistant YAML 或命令行。

## 2. 已冻结的网络约束

### 2.1 主机只使用有线网络

Home Assistant 主机统一通过以太网网线接入路由器。

本阶段明确排除：

- 主机 Wi-Fi 接入；
- 主机首次启动 Wi-Fi 热点；
- Captive Portal 配网；
- 蓝牙辅助主机配网；
- Wi-Fi 扫描、密码保存和换网恢复流程。

首启系统只需要处理：

- 有线 DHCP 获取地址；
- 无网线、无 DHCP 或网络恢复；
- 当前地址和网络状态诊断；
- Home Assistant App 的局域网发现；
- 地址变化后的服务恢复。

该约束减少约 10～20 人日的 Wi-Fi 配网与异常恢复开发工作，并降低量产和售后复杂度。

### 2.2 mDNS/服务发现边界

- Home Assistant 自身的 Zeroconf/mDNS 用于 App 实例发现和兼容设备发现；
- 主机不额外安装 Avahi，避免同机双 mDNS 栈；
- Web、SSH、MQTT 和 manager 不应仅依赖自定义 `.local` 主机名；
- 后续需提供当前 IP 显示、二维码入口或管理器发现机制，避免要求用户登录路由器查找地址。

## 3. 黄金镜像边界

### 3.1 黄金镜像应保留的公共基线

黄金镜像可以预装并保留：

- Debian 13 ARM64 运行环境；
- Docker Engine、Compose 插件和日志轮转配置；
- ZRAM、时间同步和通用系统参数；
- Home Assistant、Mosquitto、ESPHome、greenhouse-manager 等容器镜像；
- Compose 文件和 systemd 服务；
- 通用 Home Assistant 配置模板；
- 项目主题、蓝图、仪表盘模板和自动化模板；
- MQTT Discovery、节点命名和数据模型规则；
- 首启初始化、恢复出厂、升级和回滚程序；
- 工厂自检程序；
- 不含用户秘密的通用证书或公钥。

### 3.2 黄金镜像不得保留的实例数据

以下数据不能以相同内容复制到所有设备：

- Home Assistant 开发/测试管理员账号；
- Home Assistant 访问令牌、登录会话和云端身份；
- 用户家庭名称、精确位置和通知配置；
- 测试设备、实体、区域和注册表数据；
- 测试历史数据库和日志；
- MQTT 正式密码和动态安全数据库；
- greenhouse-manager 主机凭据；
- SSH 主机私钥；
- `/etc/machine-id` 最终值；
- 工厂测试节点和 retained 测试消息；
- 测试网络、FRP、Tailscale 或其他远程访问凭据；
- 任何全设备共用的默认管理密码。

这些数据不是永久删除，而是按“设备身份层、用户配置层、运行数据层”分别生成。

## 4. 数据分层模型

### 4.1 A 层：黄金镜像公共基线

所有设备相同：

- 操作系统；
- Docker 和容器镜像；
- 程序代码；
- 配置模板；
- 服务定义；
- 初始化和恢复程序。

### 4.2 B 层：每机唯一身份

在工厂首启或首次个性化阶段自动生成：

- `/etc/machine-id`；
- SSH 主机密钥；
- 产品序列号和主机 UUID；
- MQTT 主机、Home Assistant 和 manager 独立凭据；
- 本机恢复码；
- 升级授权身份；
- 安全随机种子和设备状态清单。

该阶段不要求终端用户参与。

### 4.3 C 层：用户私有配置

由最终用户拥有：

- Home Assistant Owner 账号和密码；
- 家庭名称和位置；
- 通知和远程访问配置；
- 用户授权和访问令牌。

第一版保留 Home Assistant 官方 onboarding，由用户创建自己的 Owner 账号，不预设全设备通用管理员账号。

### 4.4 D 层：运行数据

正式运行后产生：

- 节点、设备和实体注册；
- 历史数据库；
- 告警、日志和统计；
- 本机备份和异机备份；
- MQTT retained 正式状态。

## 5. 首启状态机

首启流程必须是可重复执行、可断点恢复的状态机，不能只依赖单个 `firstboot.sh` 或空标志文件。

建议状态：

```text
SEALED
  → IDENTITY_CREATED
  → SECRETS_CREATED
  → HOST_CONFIGURED
  → SERVICES_PREPARED
  → FACTORY_TESTED
  → USER_ONBOARDING_PENDING
  → POST_ONBOARDING_CONFIGURED
  → ACTIVE
```

基本要求：

- 每个阶段原子化记录开始、完成和失败原因；
- 任一阶段断电后可继续，不重复生成已承诺保持稳定的身份；
- 不得在初始化失败时回退到全设备共用凭据；
- 所有日志必须脱敏；
- 完成状态必须绑定镜像版本、配置版本和主机身份；
- 升级后可以执行增量迁移，而不是重跑完整首启流程。

建议目录：

```text
/usr/local/lib/greenhouse-firstboot/
/etc/greenhouse-firstboot/
/var/lib/greenhouse-provisioning/state.json
/var/lib/greenhouse-secrets/
/var/lib/greenhouse-factory/
```

## 6. 需要开发的模块

### M3.1 黄金镜像封装工具

建议命令：`gh-image-seal`

职责：

- 停止业务容器；
- 清除测试用户、数据库、节点、日志和 retained 测试状态；
- 删除每机必须重新生成的身份和密钥；
- 保留容器只读镜像层和通用模板；
- 扫描明文秘密和测试数据；
- 生成镜像清单、版本和 SHA-256；
- 写入 `SEALED` 状态并安全关机。

预计：3～5 人日。

### M3.2 首启供应状态机

建议服务：`greenhouse-firstboot.service`

职责：

- 判断当前状态；
- 顺序执行身份、凭据、主机配置和服务准备；
- 支持断电恢复；
- 记录结构化结果；
- 在关键条件未满足时保持失败关闭。

预计：5～8 人日。

### M3.3 每机身份和凭据生成

职责：

- 生成 machine-id、SSH 主机密钥、主机 UUID 和产品序列号；
- 生成 Mosquitto、Home Assistant、manager 的独立凭据；
- 建立最小权限文件目录和权限；
- 生成恢复码和售后识别信息；
- 验证两台克隆机之间不存在身份复用。

预计：4～7 人日。

### M3.4 Home Assistant onboarding 衔接

职责：

- 让 Home Assistant 以未创建 Owner 的干净状态启动；
- 检测用户已完成官方 onboarding；
- 不直接写入不稳定的 `.storage/auth` 内部格式；
- onboarding 长时间未完成时保持其他服务在安全状态。

预计：4～7 人日。

### M3.5 onboarding 后自动配置

建议组件：`gh-post-onboarding` 或项目自定义集成。

职责：

- 检查 Mosquitto 和 manager 状态；
- 完成或一键引导 MQTT 集成；
- 加载项目主题、仪表盘、自动化、区域和标签模板；
- 启动节点自动发现；
- 幂等创建，避免重复实体和重复配置；
- 向用户显示初始化进度和结果。

预计：7～15 人日。

### M3.6 Mosquitto 与 manager 自动供应

职责：

- 生成每机独立 Broker 管理和服务账号；
- 建立 Home Assistant 与 manager 的最小权限访问；
- 接入现有 M1/M2 MQTT Discovery、安全迁移、验证和回滚能力；
- 区分工厂测试凭据和用户正式凭据；
- 恢复出厂后重新供应；
- 保证 Home Assistant 重启后 Discovery 可恢复。

预计新增：6～12 人日。

### M3.7 有线网络初始化与诊断

职责：

- 等待 eth0 链路；
- 等待 DHCP；
- 网络超时后给出可理解状态；
- 网线插入或 DHCP 恢复后继续初始化；
- 显示当前 IP、网关、DNS 和服务入口；
- 不生成或保存任何 Wi-Fi 配置。

预计：2～4 人日。

### M3.8 恢复出厂

至少定义两级：

1. **用户数据重置**：清除 Home Assistant 用户数据、节点、历史、远程访问和用户侧凭据，保留硬件身份和工厂校准；
2. **完整安全重置**：进一步轮换 SSH、MQTT、manager 和恢复凭据。

要求：

- 断电可恢复；
- 防误触；
- 不破坏系统分区和容器镜像；
- 重置后重新进入 onboarding；
- 支持从备份恢复。

预计：7～12 人日。

### M3.9 工厂烧录与自检工具

工厂流程：

```text
烧录黄金镜像
→ 扩展分区
→ 启动每机个性化
→ 检查存储、网卡、MAC、时间和 Docker
→ 检查 Home Assistant、Mosquitto、manager
→ 验证身份唯一性
→ 生成设备编号和测试记录
→ 安全关机
```

输出结构化工厂记录，不在日志中包含明文秘密。

预计：6～12 人日。

### M3.10 健康监控、升级和回滚接入

职责：

- Home Assistant HTTP、Mosquitto 和 manager 健康检查；
- 磁盘、内存、温度和数据库增长监控；
- 升级前备份；
- 新旧镜像空间检查；
- 更新失败自动回滚；
- 首启状态和配置 schema 迁移；
- 接入现有 manager 生产执行和回滚审计链。

预计：8～15 人日。

## 7. 分阶段范围和工程量

### M3-A：最小可用黄金镜像

范围：

- 制镜清理；
- 每机身份和凭据；
- 基础首启状态机；
- 有线 DHCP；
- Home Assistant 官方 onboarding；
- 容器自动启动。

预计：15～25 人日。

适用：研发重建和内部验证。

### M3-B：受控试产版

在 M3-A 基础上增加：

- Mosquitto、manager 自动供应；
- onboarding 后项目初始化；
- 节点和仪表盘自动创建；
- 恢复出厂；
- 工厂自检；
- 基础健康监控。

累计预计：35～50 人日。

适用：少量知情测试用户和受控试产。

### M3-C：正式量产版

在 M3-B 基础上增加：

- 所有关键阶段断电恢复；
- 完整升级、回滚和备份恢复；
- 安全审计和秘密泄露扫描；
- 生产记录和批量烧录工具；
- 多批次硬件兼容；
- 7～30 天稳定性测试；
- 完整售后恢复流程。

累计预计：70～100 人日。

该估算已排除主机 Wi-Fi 配网工作；测试、修复和文档通常占总工程量的 35%～50%。

## 8. 用户交付流程目标

第一版正式目标：

```text
插入网线
→ 接通电源
→ 主机自动完成每机个性化
→ 手机 App 自动发现 Home Assistant
→ 用户创建自己的管理员账号
→ 系统自动完成项目初始化
→ 监测节点自动出现
→ 仪表盘自动可用
```

允许保留的用户操作只有：

- 接入网线；
- 创建自己的 Home Assistant 管理员账号；
- 必要时确认家庭名称、地区和通知权限。

不得要求用户：

- SSH 登录；
- 执行 Docker 命令；
- 配置 MQTT；
- 修改 YAML；
- 查找或编辑系统密钥；
- 理解 Linux、Docker 或 Home Assistant 内部结构。

## 9. 验收重点

至少覆盖：

- 两台以上克隆机同时启动，machine-id、SSH 密钥、主机 UUID、MQTT 客户端 ID 和正式凭据均不同；
- 首启每个关键阶段断电后可继续；
- 无网线启动后，插入网线可继续；
- 无 DHCP 时不误标记初始化完成；
- Home Assistant 启动慢时不误判失败；
- 用户长期未完成 onboarding 时不暴露默认管理凭据；
- 重复执行首启程序不会重复创建身份、账号、集成或仪表盘；
- 恢复出厂不会删除黄金镜像和硬件校准信息；
- 升级失败可以恢复上一可运行版本；
- 工厂测试数据和正式用户数据之间无残留；
- 日志、工厂记录和 GitHub 仓库中无明文秘密。

## 10. 当前决策

1. 同意采用“黄金公共镜像 + 每机首启个性化 + 用户 onboarding 后自动初始化”的量产路线；
2. 将该工作定义为独立的 M3 产品化开发阶段；
3. 第一阶段不追求绕过 Home Assistant 官方 Owner 创建流程；
4. 主机统一使用有线以太网，不开发主机 Wi-Fi 配网；
5. 优先完成 M3-A，再在试产验证基础上推进 M3-B 和 M3-C；
6. 现有 M1/M2 的 MQTT Discovery、安全迁移、生产执行和回滚能力应作为 M3 的依赖复用，而不是重新实现。
