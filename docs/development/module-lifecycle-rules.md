# 模块生命周期规则

本规则适用于 host/greenhouse-manager、firmware/esphome_rc 等所有由 AI 助手
迭代开发的代码包。目的：避免历史迭代中出现孤立模块、重复版本、
测试通过但从未接入生产路径的代码。

## 新建文件前

- 先搜索是否已有职责相同或相近的模块。如果有，优先编辑/重构已有模块，
  而不是新建一个变体文件。
- 禁止用版本号或阶段名作为文件名后缀（如 `_v2`、`_v69`、`_lab`、`_prepare`、
  `_probe`、`stageXdY`）。迭代过程用 git commit 记录，不用文件名记录。

## 完成定义（Definition of Done）

一个任务只有满足以下条件才算完成：
- 新代码被某个现有入口点（pyproject.toml 中的 CLI，或 app 主流程）
  实际调用到，而不仅仅是被单元测试导入。
- 有至少一个集成测试验证这条调用链，单元测试通过不足以证明"完成"。
- 如果任务的产出是探索性/实验性质，不接入生产入口点，必须放在
  `experiments/`（或对应的 `_lab`/`board_lab` 隔离目录），不得进入
  `src/greenhouse_manager/` 或 `firmware/esphome_rc/components/` 正式目录。

## 替换已有实现时

- 如果新代码是为了替换某个旧模块，必须在同一个 PR 里删除旧模块
  （及其专属测试），不允许新旧版本同时留在代码库里"以防万一"。
- 如果不确定旧模块是否还有依赖方，先做依赖可达性检查
  （从 pyproject.toml 定义的入口点出发做传递依赖分析），
  确认后再删除。

## protocols/ 目录约束

- `protocols/` 只能存放跨固件、主机、网关的冻结接口定义（topic、
  payload、schema）。
- 一次性的运维操作记录、生产授权记录、迁移执行日志等，一律放入
  `docs/handoffs/` 或 `docs/status/`，不得放在 `protocols/` 下。

## 定期健康检查

- 每完成一个里程碑，或每月一次，运行一次模块可达性分析
  （从入口点出发标记可达/不可达模块），并与上一次基线对比孤立模块数量。
- 如果孤立模块数量明显上升，视为需要立即处理的技术债，
  而不是拖到下次大清理。
