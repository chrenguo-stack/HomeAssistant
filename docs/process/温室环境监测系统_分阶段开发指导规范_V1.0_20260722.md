# 温室环境监测系统
## 分阶段开发指导规范（修正版）V1.0

**文档版本：** V1.0  
**文档性质：** 研发流程、GitHub 使用、验证与交付指导规范  
**适用项目：** 温室环境监测系统（ESP32-C6、greenhouse-manager、Home Assistant 主机、M401A/T1）  
**编制日期：** 2026-07-22  
**制定背景：** 在不降低安全、可追溯、可回滚要求的前提下，减少 GitHub 往返、重复脚本、候选链重建和阶段碎片化造成的时间消耗。  
**起草基线：** `main=b8cc4f68d29393cdf9da7d00fdfeef28ee147c7b`。该 SHA 仅表示本文档起草时的仓库基线，不构成任何产品候选、固件候选或实机授权。

## 1. 目的与适用边界

本规范用于统一后续各阶段的开发、测试、GitHub 协作、候选冻结、实机验收、证据保存和交接方式，重点解决以下问题：

1. 小修正触发整条候选与证据链重复重建；
2. 过早推送导致 GitHub CI 反复暴露本地即可发现的问题；
3. 每个故障新建近似脚本、近似分支，形成碎片化；
4. 运行时修补、源码、候选镜像、现场环境之间身份不一致；
5. 日志、二进制和中间证据大量进入 Git 历史；
6. 新对话反复恢复仓库状态、基线和下一步门禁；
7. 需要人工决策和授权的节点不清晰，造成频繁中断。

本规范不替代：

- 产品架构和阶段路线文档；
- 协议规范、ADR、故障矩阵和测试协议；
- 无线电、EMC、强电、安全生产等合规要求；
- 针对某次实机测试明确给出的临时禁令或更严格约束。

发生冲突时，优先级从高到低为：

1. 用户在当前阶段明确下达的安全边界和禁止事项；
2. 已冻结的协议、产品架构和阶段验收标准；
3. 本规范；
4. 普通开发习惯。

## 2. 总体原则

### 2.1 安全标准不因效率优化而降低

以下要求继续保持：

- 生产写操作默认关闭；
- 未经明确授权不得连接真实 Broker、写入物理 NVS、烧录实板、读取或烧写 eFuse；
- 未经授权不得修改 M401A、T1、Home Assistant、Mosquitto、greenhouse-manager 生产环境；
- 一次性人工授权必须与目标、generation、候选哈希、有效期和回滚点绑定；
- 每次实机修改必须具备清理路径和回滚路径；
- 不得用 runtime overlay 代替正式源码固化；
- 不得重放旧授权，不得把历史证据冒充当前候选证据。

### 2.2 先降低失败成本，再增加验证强度

验证顺序固定为：

> 静态检查 → 单元测试 → host 模型 → compile-only → 本地集成测试 → GitHub CI → 不可变候选 → 隔离验收 → 实机验收 → 发布。

越靠后的步骤成本越高，越不得用于发现本地可提前发现的问题。

### 2.3 一个阶段只维护一条主开发线

- 每个正式阶段创建一个唯一命名分支；
- 同一阶段内的普通缺陷修复继续提交到该分支，不为每个错误新建分支；
- 只有发生“基线改变、阶段目标改变、已冻结候选必须保留、需要并行隔离”时，才创建后继分支；
- 已冻结的历史阶段分支不得复用或改写。

### 2.4 候选冻结必须晚于功能和本地验证冻结

候选构建不得在以下事项尚未完成时启动：

- 需求和接口仍频繁变化；
- 本地 preflight 未通过；
- host 模型或 compile-only 仍失败；
- 测试协议和故障矩阵尚未稳定；
- 预计还会修改源文件、构建定义或依赖版本。

### 2.5 Git 保存“可复现定义”，Artifact 保存“大体积证据”

Git 中保存：

- 源码；
- 配置模板；
- 测试代码；
- 协议和 ADR；
- 候选 manifest；
- 哈希、摘要、结论和证据索引；
- 可复现命令和清理说明。

GitHub Actions Artifact 或发布存档保存：

- 编译产物；
- 完整日志；
- 大型 JSON 证据；
- 临时诊断包；
- 实机采样原始数据；
- 可由源码重新生成的中间文件。

## 3. 开发工作分级

每项任务开始前必须归类，避免所有任务都走最高成本流程。

### 3.1 A 类：文档与非执行性变更

包括：

- 路线文档、ADR、设计说明、交接文档；
- 注释、README、测试协议草案；
- 不影响构建和运行的格式修正。

最低门禁：

- Markdown/链接/格式检查；
- 文档内部一致性检查；
- Draft PR；
- 无需生成产品候选和实机授权。

### 3.2 B 类：模型、工具和 compile-only 变更

包括：

- host 模型；
- mock adapter；
- CI、测试框架和证据格式；
- 不接触真实环境的 compile-only 固件；
- 只读诊断工具。

最低门禁：

- 本地 preflight；
- 单元测试和故障注入；
- GitHub CI；
- 不得默认触发真实网络或写操作。

### 3.3 C 类：产品源码和运行行为变更

包括：

- 固件正式组件；
- greenhouse-manager 正式源码；
- MQTT、身份、授权、NVS、OTA、容器运行行为；
- 部署和恢复工具。

必须经过完整阶段门禁，并在候选冻结后才允许进入实机验收。

### 3.4 D 类：生产或不可逆操作

包括：

- 真实 Broker 认证切换；
- 生产容器重建；
- 物理 NVS 写入；
- 实板烧录；
- eFuse 操作；
- 黄金镜像制作或批量写入。

必须具备：

- 明确用户授权；
- generation-bound 一次性令牌；
- 目标候选哈希；
- 预检证据；
- 回滚和清理方案；
- 操作后复核；
- 不可逆操作的二次确认。

## 4. 标准阶段生命周期

所有新阶段默认采用 S0 至 S8。阶段可以合并，但不得跳过对应门禁含义。

## S0：任务接收与基线确认

### 进入条件

- 已收到阶段目标或问题描述。

### 必做事项

1. 确认仓库、默认分支和起始 SHA；
2. 确认当前阶段编号和上阶段冻结结果；
3. 确认禁止操作和允许操作；
4. 确认是否允许联网、实机、生产写入；
5. 检查是否已有同目标分支、PR、脚本或测试框架；
6. 创建或更新阶段状态文件。

### 退出条件

形成最小阶段头信息：

```text
STAGE=
BASE_SHA=
BRANCH=
SCOPE=
PROHIBITED=
ALLOWED=
REQUIRED_GATES=
USER_DECISIONS_PENDING=
```

### GitHub 原则

本阶段只做读取和状态核对；除创建阶段分支外，不应产生产品代码提交。

---

## S1：范围冻结与验收设计

### 必做事项

1. 把目标拆为“必须完成、明确不做、后续再做”；
2. 冻结接口变更范围；
3. 先写测试协议、故障矩阵和证据字段，再写实现；
4. 确定清理与回滚方法；
5. 确定哪些结论可由模型证明，哪些必须实机证明；
6. 确定人工决策门。

### 退出条件

- 任务不存在未定义的写操作；
- 验收条件可编码、可观察、可复核；
- 候选物类型和证据格式已明确。

### 决策门 D1

只有以下情况需要打断用户：

- 产品行为存在两个以上不可兼容方案；
- 会改变冻结接口或硬件假设；
- 会新增不可逆风险；
- 需要扩大阶段范围。

普通实现细节不应频繁询问。

---

## S2：本地实现

### 开发规则

- 在阶段唯一主分支上持续开发；
- 优先扩展现有模块、fixture 和测试工具；
- 禁止为每次失败复制一份近似脚本；
- 正式逻辑必须进入源码，runtime overlay 只允许作为临时诊断，并不得成为验收结果；
- 写操作入口统一采用 `disabled_by_default`；
- 所有环境依赖通过显式参数传入，不隐式读取生产路径。

### 提交策略

允许本地多个小提交，但推送前应整理为有意义的逻辑提交。推荐类型：

- `feat(scope): ...`
- `fix(scope): ...`
- `test(scope): ...`
- `docs(scope): ...`
- `refactor(scope): ...`

不得使用“try again”“fix2”“final-final”等不可追踪提交说明。

### 退出条件

- 代码、测试、协议和清理流程同步更新；
- 不存在已知的本地失败；
- 无真实环境依赖泄漏到默认测试路径。

---

## S3：统一本地 Preflight

### 目标

在推送 GitHub 前发现绝大多数低成本问题。

### 统一入口

每个主要子系统应逐步收敛为单一命令，例如：

```bash
./tools/preflight.sh --stage H3-N2-2D7 --profile local-safe
```

或：

```bash
python -m tools.preflight --stage H3-N2-2D7 --profile local-safe
```

### Preflight 最低内容

1. 格式与静态检查；
2. 单元测试；
3. host 模型和故障注入；
4. compile-only；
5. 禁止生产地址、密码、令牌和绝对路径检查；
6. 变更文件与阶段范围匹配检查；
7. 脚本名称冲突和重复实现检查；
8. manifest schema 检查；
9. 清理流程 dry-run；
10. 当前工作树干净性检查。

### 输出要求

只输出一个机器可读总结和一个人类可读摘要：

```json
{
  "schema": "gh.dev.preflight/1",
  "stage": "H3-N2-2D7",
  "base_sha": "...",
  "head_sha": "...",
  "status": "passed",
  "checks": {},
  "production_io_attempted": false,
  "writes_performed": false
}
```

### 退出条件

- `status=passed`；
- 工作树无未记录变更；
- 可以明确说明 GitHub CI 预计验证什么，而不是“推上去看看”。

## S4：GitHub CI 批量验证

### 推送原则

- 本地 preflight 未通过，不推送；
- 将相关修正批量完成后再推送，避免每改一行触发一次全量 CI；
- Draft PR 可提前建立用于集中讨论，但完整 CI 只在 `ci-ready` 状态或等价条件下执行；
- 对文档、固件、主机端使用 path filter，避免无关工作流运行；
- 同分支新提交应取消旧的排队或运行中工作流；
- 失败后优先重跑失败 job，不默认重跑全部成功 job。

### CI 分层

建议分为：

1. **Fast Gate**：格式、静态检查、关键单元测试；
2. **Model Gate**：host 模型、故障矩阵、协议一致性；
3. **Build Gate**：Docker、ESPHome/ESP-IDF compile-only；
4. **Evidence Gate**：manifest、哈希、证据 schema；
5. **Candidate Gate**：仅在候选冻结后启用。

### CI 失败处理

- 第一时间判断属于代码失败、环境失败还是偶发失败；
- 偶发失败允许重跑单个 job；
- 可重复失败必须本地复现后修复；
- 不得通过扩大重试次数掩盖确定性缺陷；
- 不得为通过 CI 而删除必要断言。

### 退出条件

- 所有适用 gate 通过；
- PR 中能够区分“已证明”和“尚待实机证明”的结论；
- CI artifact 可定位并与 head SHA 绑定。

---

## S5：候选冻结与不可变构建

### 启动条件

同时满足：

- 范围冻结；
- 源码冻结；
- 本地 preflight 通过；
- GitHub CI 通过；
- 测试协议和故障矩阵冻结；
- 不再预计修改产品行为。

### 候选 generation

每个候选使用单调递增 generation：

```text
STAGE 2D-7 / CANDIDATE G1
STAGE 2D-7 / CANDIDATE G2
```

候选失败后：

- 保留失败候选的 manifest 和失败摘要；
- 修正源码后产生新 SHA 和新 generation；
- 不覆盖旧候选；
- 不必为同阶段每个 generation 新建分支，除非旧分支已被明确冻结为历史只读分支。

### 候选 manifest 最低字段

```json
{
  "schema": "gh.candidate/1",
  "stage": "H3-N2-2D7",
  "generation": 1,
  "repository": "chrenguo-stack/HomeAssistant",
  "source_sha": "...",
  "base_sha": "...",
  "toolchain": {},
  "build_inputs": {},
  "artifact_sha256": {},
  "tests": {},
  "writes_enabled": false,
  "production_environment_used": false,
  "rollback_contract": "..."
}
```

### 退出条件

- 候选物不可变；
- 源码、配置、工具链和 artifact 哈希闭环；
- 后续授权只能指向该 generation；
- 任何修改都会自动使候选失效。

---

## S6：隔离验收与实机验收

### S6A：隔离验收

优先使用：

- `network=none`；
- 临时目录；
- mock Broker；
- 临时 NVS 镜像；
- 容器临时网络；
- 非生产端口；
- compile-only 或 host 仿真。

必须验证：

- 默认不写；
- 拒绝未授权写入；
- generation 不匹配时失败关闭；
- 中断后可清理；
- 失败后无残留服务或临时凭据。

### S6B：实机验收

只有隔离验收不能证明的项目才进入实机。

### 决策门 D2

需要用户明确授权的内容：

- 连接哪台设备；
- 读取还是写入；
- 目标候选和 generation；
- 有效期；
- 允许修改的对象；
- 回滚点；
- 失败时是否自动回滚。

### 实机命令交付要求

需要用户执行时，必须提供完整命令，不省略：

- 本地文件绝对路径；
- `scp` 目标；
- `ssh` 命令；
- 校验命令；
- 执行命令；
- 预期输出；
- 清理命令。

`.py` 和 `.sh` 文件不得复用历史文件名；但应优先使用稳定的参数化框架，只为本次授权生成唯一入口或请求文件，避免复制整套实现。

### 退出条件

- 验收结果与候选 generation 绑定；
- 生产服务修改范围可证明；
- 回滚或清理结果已复核；
- 失败不得被描述为通过。

## S7：合并、发布与归档

### 合并前检查

- Draft PR 中所有门禁结论已更新；
- 代码与测试同步；
- 文档和 ADR 同步；
- 候选 manifest 和证据索引已提交；
- 大型日志和二进制未进入 Git；
- 不含生产秘密、用户路径、真实密码或临时授权；
- 已说明兼容性、迁移和回滚。

### 决策门 D3

以下情况必须由用户决定是否接受：

- 已知但未解决的风险；
- 降级验收；
- 基线缺口豁免；
- 兼容性破坏；
- 未完成的实机覆盖。

不得由开发流程自动默认为接受。

### 发布原则

- 文档或测试框架可以独立合并；
- 产品候选只有在适用门禁全部通过后才可转正式；
- PR 默认保持 Draft，直到用户或阶段规则明确允许 Ready；
- 不自动合并重大安全、身份、凭据、NVS、OTA 和生产部署变更。

## S8：阶段关闭与交接

### 必须输出

1. 阶段目标与完成度；
2. 起始 SHA、最终 SHA、分支、PR；
3. 已通过 gate；
4. 未通过或未执行 gate；
5. 候选 generation 和 artifact 哈希；
6. 实机修改与回滚状态；
7. 当前生产环境是否被修改；
8. 遗留问题；
9. 下一阶段唯一建议入口；
10. 下一轮对话提示词。

### 状态文件

建议每个活动阶段维护：

```text
docs/status/STAGE_STATUS_<stage>.md
```

活动阶段只保留一个权威状态文件。交接文档是阶段快照，不应与状态文件产生两套相互冲突的“当前状态”。

## 5. 分支、PR 与提交规范

### 5.1 分支命名

推荐：

```text
feature/<system>-<stage>-<short-purpose>-<date>
fix/<system>-<stage>-<short-purpose>-<date>
docs/<short-purpose>-v<version>-<date>
```

要求：

- 包含阶段或目的；
- 唯一且可读；
- 不使用 `test2`、`new-final` 等名称；
- 不复用历史冻结分支；
- 不在无关阶段分支上顺带开发新功能。

### 5.2 一阶段一主分支

默认规则：

- 阶段开始：创建一次分支；
- 阶段开发：普通修正继续在该分支；
- 候选 generation 更新：仍在该分支产生新提交；
- 阶段结束：合并或关闭并冻结；
- 新阶段：从指定新基线创建新分支。

### 5.3 Draft PR 作为阶段控制面

Draft PR 应包含：

- 目标；
- 明确不做事项；
- 基线 SHA；
- gate 清单；
- 风险；
- 候选 generation；
- 实机状态；
- artifact 链接；
- 用户决策项。

不要把 PR 评论当作唯一状态来源，关键状态必须同步回仓库文档或 manifest。

## 6. 脚本与测试框架优化

### 6.1 禁止“复制—改名—再复制”成为默认开发方式

优先结构：

```text
tools/framework/
  authorization.py
  candidate.py
  evidence.py
  rollback.py
  remote_exec.py

tools/entrypoints/
  <stage-unique-entry>.py

tests/
  fixtures/
  matrices/
```

稳定框架文件可持续复用；阶段入口、授权请求和实机执行文件保持唯一命名。

### 6.2 新建脚本前检查

必须回答：

1. 是否已有同类功能；
2. 能否通过参数或 fixture 扩展；
3. 新脚本是否只改了常量、路径或阶段名；
4. 是否应改为数据驱动测试；
5. 是否会增加另一套证据 schema。

仅当职责确实不同或旧脚本已冻结不可修改时，才新建完整脚本。

### 6.3 文件命名

建议包含：

```text
<system>_<stage>_<purpose>_<generation>_<short-sha>.py
```

唯一命名用于防止误执行，不等于必须复制实现。

## 7. GitHub CI 成本控制规范

### 7.1 必须逐步落实

- `paths` / `paths-ignore`；
- `concurrency` + `cancel-in-progress`；
- 依赖和构建缓存；
- 快速门与重型门分离；
- 仅候选标签或显式输入触发候选构建；
- artifact 设定合理保留期限；
- 支持重跑失败 job；
- compile-only 与真实部署完全分离。

### 7.2 推送节奏

建议每个阶段控制为：

- 首次骨架推送；
- 本地功能完成后的 CI-ready 推送；
- CI 修正推送；
- 候选冻结推送；
- 实机证据/文档收尾推送。

不把该数量作为硬限制，但偏离明显时应检查是否存在碎片化。

### 7.3 禁止事项

- 以 GitHub CI 代替本地基本测试；
- 在每次小编辑后触发全部 Docker 和固件构建；
- 为偶发失败反复提交无意义变更；
- 将大日志提交到仓库仅为保留证据；
- 同时维护多个含义相同的工作流。

## 8. 证据、Artifact 与可追溯性

### 8.1 证据分三层

**L1 摘要证据：** 结论、状态、关键哈希，进入 Git。  
**L2 结构化证据：** 完整 JSON、测试矩阵结果，进入 Artifact，并在 Git 中保存索引和哈希。  
**L3 原始证据：** 完整日志、镜像、抓包、长周期采样，进入受控归档，不直接进入普通 Git 历史。

### 8.2 证据必须绑定

- repository；
- source SHA；
- stage；
- generation；
- tool version；
- target identity；
- execution time；
- authorization identity；
- artifact SHA256。

### 8.3 不确定性表达

只能使用：

- `passed`：证据完整且门禁通过；
- `failed`：门禁失败；
- `not_run`：未执行；
- `inconclusive`：执行过但证据不足；
- `waived`：经明确人工决策接受缺口。

不得把 `not_run` 或 `inconclusive` 写成“基本通过”。

## 9. 用户决策与中断规则

除非用户另有要求，开发过程中仅在以下节点中断：

1. **D1 架构/范围决策**：存在不可兼容方案；
2. **D2 实机/生产授权**：将接触真实设备或写操作；
3. **D3 风险豁免**：需要接受未闭环缺口；
4. **D4 发布决策**：Ready、合并、正式发布或不可逆动作。

以下事项不应频繁中断：

- 普通代码结构；
- 测试 fixture 选择；
- 文档排版；
- 不改变产品行为的重构；
- 本地模型内部实现。

用户要求“持续推进”时，应一直推进到下一个真正决策门，而不是在每个 CI 或小修正后停下。

## 10. 阶段效率指标

每个阶段结束时记录以下指标，用于发现流程问题，不用于追求表面数字：

```text
PUSH_COUNT=
CI_RUN_COUNT=
CI_FAILED_RUN_COUNT=
FAILURES_FOUND_LOCALLY=
FAILURES_FIRST_FOUND_IN_CI=
CANDIDATE_GENERATION_COUNT=
BRANCH_COUNT=
NEW_SCRIPT_COUNT=
REUSED_FRAMEWORK_COUNT=
REAL_MACHINE_RUN_COUNT=
ROLLBACK_COUNT=
```

建议关注：

- CI 首次发现的低级错误是否下降；
- 每阶段分支是否接近 1；
- 新增完整脚本数量是否下降；
- 候选 generation 是否因过早冻结而过多；
- 实机运行是否集中验证模型无法证明的事项；
- GitHub 等待时间是否从主要瓶颈变为最终复核环节。

## 11. 渐进式落地计划

### 第一步：立即执行

- 新阶段采用“一阶段一主分支”；
- 建立单一 `STAGE_STATUS`；
- 推送前必须完成本地 preflight；
- PR 默认 Draft；
- 大日志改存 Artifact；
- 候选构建延后到 CI 全部通过后。

### 第二步：未来 1—2 个阶段内完成

- 收敛统一 preflight 入口；
- 抽取授权、证据、回滚共用框架；
- 设置 Actions concurrency 和 path filter；
- 将故障矩阵改为数据驱动；
- 建立候选 manifest schema V1。

### 第三步：流程稳定后完成

- 候选构建与授权请求自动绑定；
- 自动生成阶段状态摘要；
- 自动生成下一轮交接提示词；
- 对 M401A/T1/节点固件形成统一但分目标的验收框架；
- 定期删除失效 Artifact，保留长期发布证据。

### 迁移原则

- 不追溯性重写已完成历史阶段；
- 不修改已冻结分支和证据；
- 从下一个新阶段开始执行；
- 当前进行中的阶段可选择性采用状态文件、preflight 和 Artifact 优化，但不得因此改变其既定安全边界。

## 13. 本规范的成功标准

本规范实施后，应出现以下变化：

- GitHub CI 主要承担独立复核，而不是基础调试；
- 同阶段分支和脚本数量明显下降；
- 候选生成更晚、generation 数量减少；
- 实机测试次数减少但证据质量提高；
- 生产安全边界保持不变；
- 新对话能够通过状态文件和提示词快速恢复；
- 项目开发时间更多用于产品功能、故障模型和用户价值，而不是重复整理 GitHub 状态。

本规范属于可演进流程文件。若连续两个阶段的效率指标显示某项门禁成本高但不能发现有效问题，应形成 ADR 或修订版说明后调整，不得在单次开发中临时绕过。
