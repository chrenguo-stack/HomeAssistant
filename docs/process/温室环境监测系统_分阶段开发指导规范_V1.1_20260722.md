**文档性质：** 研发流程、GitHub 使用、协作分工、验证与交付指导规范  
**适用项目：** 温室环境监测系统（ESP32-C6、greenhouse-manager、Home Assistant 主机、M401A/T1）  
**编制日期：** 2026-07-22  
**起草基线：** `main=b8cc4f68d29393cdf9da7d00fdfeef28ee147c7b`  
**替代关系：** 本文件替代同分支中的 V1.0 草案。

> 本规范的目标是在不降低安全、可追溯、可回滚要求的前提下，减少 GitHub 往返、重复脚本、候选链重建、阶段碎片化和用户等待时间。

## 0. V1.1 修订摘要

V1.1 在 V1.0 的阶段开发规范基础上，新增并冻结以下协作机制：

1. 每个阶段同时维护“助手开发队列”和“用户操作队列”；
2. 将长编译、只读环境采集、物理检查、长周期观测和已授权实机操作交给用户批量执行；
3. 禁止采用“一条命令—一段输出—再给下一条命令”的低效率交互作为默认方式；
4. 用户操作必须通过完整、可校验、可停止、可清理的批量测试包交付；
5. 用户只承担机械执行、现场观察和产品决策，不承担协议设计、故障归因或安全模型设计；
6. 新增用户等待、批量测试包和并行任务效率指标；
7. 下一轮对话提示词默认要求主动拆分可并行的用户操作任务。

## 1. 目的、边界与优先级

本规范统一后续各阶段的：

- 开发与测试顺序；
- GitHub 分支、PR、提交和 CI 使用方式；
- 候选冻结、授权、实机验收和回滚方式；
- 用户与助手的职责分工；
- 证据、Artifact、阶段状态和交接方式。

重点解决：

- 小修正触发整条证据链重建；
- GitHub CI 被当作基础调试环境；
- 近似脚本、分支和候选物快速增殖；
- 源码、运行时修补、构建产物和实机身份不一致；
- 用户长时间等待助手输出，而可并行任务没有提前交付；
- 实机测试被拆成大量串行的小命令；
- 新对话反复恢复基线和当前状态。

本规范不替代产品架构、冻结协议、ADR、测试协议、故障矩阵、无线电/EMC/强电合规要求，也不覆盖当前阶段另行下达的更严格禁令。

发生冲突时，优先级为：

1. 当前阶段明确安全边界与禁止事项；
2. 已冻结产品架构、协议和验收标准；
3. 本规范；
4. 普通开发习惯。

## 2. 总体原则

### 2.1 安全标准不因效率优化而降低

必须持续满足：

- 所有生产写操作默认关闭；
- 未经明确授权不得连接真实 Broker、写入物理 NVS、烧录或运行实板、读取或烧写 eFuse；
- 未经授权不得修改 M401A、T1、Home Assistant、Mosquitto 或 greenhouse-manager 生产环境；
- 一次性人工授权必须与目标、generation、候选哈希、有效期和回滚点绑定；
- 每次实机修改都必须有停止、清理和回滚路径；
- 不得用 runtime overlay 代替正式源码固化；
- 不得重放旧授权或把历史证据冒充当前候选证据；
- 不得要求用户在真实设备上运行未经审查、无哈希、无边界的临时命令。

### 2.2 先降低失败成本，再提高验证强度

默认验证顺序：

> 静态检查 → 单元测试 → host 模型 → compile-only → 本地集成测试 → GitHub CI → 不可变候选 → 隔离验收 → 实机验收 → 发布。

越靠后的步骤成本越高，越不得用于发现本地可提前发现的问题。

### 2.3 一阶段一条主开发线

- 每个正式阶段创建一个唯一命名分支；
- 普通缺陷修复继续提交到该分支；
- 不为每个 CI 错误或 generation 新建分支；
- 只有基线改变、阶段目标改变、并行隔离或已冻结候选必须保留时才建后继分支；
- 历史冻结分支不得复用或改写。

### 2.4 候选冻结晚于功能与验证冻结

以下任一条件存在时，不得生成正式候选：

- 需求或接口仍频繁变化；
- 本地 preflight 未通过；
- host 模型、故障矩阵或 compile-only 仍失败；
- 测试协议、证据格式或清理流程尚未稳定；
- 预计仍会修改产品行为、构建定义或依赖版本。

### 2.5 Git 保存可复现定义，Artifact 保存大型证据

Git 保存源码、配置模板、测试、协议、ADR、manifest、哈希、摘要、证据索引和可复现命令。

CI Artifact 或受控归档保存编译产物、完整日志、大型 JSON、抓包、镜像、长周期采样和临时诊断包。

## 3. 开发工作分级

### A 类：文档与非执行性变更

包括路线、ADR、设计说明、交接文档、README 和不影响运行的格式修正。

最低门禁：文档一致性检查、Draft PR；不生成产品候选，不需要实机授权。

### B 类：模型、工具和 compile-only 变更

包括 host 模型、mock adapter、CI、测试框架、证据格式、compile-only 固件和只读诊断工具。

最低门禁：本地 preflight、单元测试、故障注入、适用 CI；默认不得触发真实网络或写操作。

### C 类：产品源码与运行行为变更

包括固件正式组件、greenhouse-manager 正式源码、MQTT、身份、授权、NVS、OTA、容器运行行为、部署和恢复工具。

必须经过完整阶段门禁，并在候选冻结后才允许进入实机验收。

### D 类：生产或不可逆操作

包括真实 Broker 认证切换、生产容器重建、物理 NVS 写入、实板烧录、eFuse、黄金镜像和批量写入。

必须具备明确授权、generation-bound 一次性令牌、候选哈希、预检证据、回滚/清理方案、操作后复核；不可逆动作需二次确认。

## 4. 标准阶段生命周期 S0—S8

阶段可以合并，但不得跳过相应门禁含义。

### S0：任务接收与基线确认

必须确认：仓库、默认分支、起始 SHA、阶段编号、允许/禁止操作、联网/实机/生产权限、已有分支/PR/框架，以及用户可并行执行的任务。

阶段状态至少包含：

```text
STAGE=
BASE_SHA=
BRANCH=
SCOPE=
PROHIBITED=
ALLOWED=
REQUIRED_GATES=
USER_DECISIONS_PENDING=
ASSISTANT_QUEUE=
USER_OPERATION_QUEUE=
```

S0 只做读取和状态核对；除创建阶段分支外，不产生产品代码提交。

### S1：范围冻结与验收设计

必须先完成：

1. “必须完成、明确不做、后续再做”拆分；
2. 接口范围冻结；
3. 测试协议、故障矩阵和证据字段；
4. 清理与回滚方法；
5. 模型可证明项与实机必证项划分；
6. 用户操作队列和批量测试包草案；
7. 人工决策门。

只有产品行为存在不可兼容方案、改变冻结接口/硬件假设、新增不可逆风险或扩大范围时，触发 D1 架构/范围决策。

### S2：本地实现

- 在阶段唯一主分支持续开发；
- 优先扩展已有模块、fixture 和参数化测试工具；
- 禁止为每次失败复制近似脚本；
- 正式逻辑进入源码；runtime overlay 只能临时诊断；
- 写入口统一 `disabled_by_default`；
- 环境依赖必须显式传参，不隐式读取生产路径；
- 能与助手开发并行的用户只读采集、长编译和物理检查应在本阶段尽早交付，而不是等源码全部完成后再提出。

### S3：统一本地 Preflight

主要子系统应收敛为单一入口，例如：

```bash
./tools/preflight.sh --stage H3-N2-2D7 --profile local-safe
```

最低内容：格式/静态检查、单元测试、host 模型、故障注入、compile-only、秘密/生产地址/绝对路径检查、范围检查、重复脚本检查、manifest schema、清理 dry-run 和工作树状态检查。

输出一个机器可读总结和一个人类可读摘要；必须明确 `production_io_attempted=false`、`writes_performed=false`。

本地 preflight 未通过，不推送 GitHub。

### S4：GitHub CI 批量验证

- 将相关修正批量完成后再推送；
- Draft PR 可提前建立，但重型 CI 仅在 `ci-ready` 或等价状态运行；
- 使用 path filter、concurrency、cancel-in-progress 和缓存；
- Fast、Model、Build、Evidence、Candidate Gate 分层；
- 偶发失败可重跑单个 job；可重复失败必须本地复现；
- 不得扩大重试次数掩盖确定性缺陷。

### S5：不可变候选冻结

只有范围、源码、测试协议、故障矩阵、本地 preflight 和适用 CI 全部冻结后才能启动。

候选使用单调递增 generation；失败候选保留 manifest 和失败摘要，不覆盖旧候选，不必为每个 generation 新建分支。

manifest 至少绑定 repository、base/source SHA、stage、generation、工具链、构建输入、artifact SHA256、测试结果、写入状态和回滚合同。

### S6：隔离与实机验收

S6A 优先使用 `network=none`、临时目录、mock Broker、临时 NVS、临时容器网络、非生产端口和 host 仿真。

必须验证默认不写、未授权写入被拒、generation 不匹配时失败关闭、中断可清理、失败后无残留服务或临时凭据。

只有隔离环境无法证明的事项才进入 S6B 实机验收。

实机操作触发 D2 授权，必须明确设备、读/写性质、候选 generation、有效期、允许修改对象、回滚点和失败策略。

### S7：合并、发布与归档

合并前检查：代码/测试/文档同步，manifest 与证据索引完整，大日志和二进制未进入 Git，不含秘密和真实用户路径，兼容性、迁移与回滚已说明。

已知风险、降级验收、基线缺口、兼容性破坏或未完成实机覆盖触发 D3 风险豁免。

PR 默认保持 Draft；Ready、合并、正式发布或不可逆动作触发 D4 发布决策。

### S8：阶段关闭与交接

必须记录：目标完成度、起始/最终 SHA、分支/PR、已通过/未运行/失败 gate、候选 generation、artifact 哈希、实机与回滚状态、生产环境修改状态、遗留问题、下一阶段唯一入口、下一轮提示词，以及用户操作队列完成情况。

活动阶段只维护一个权威 `docs/status/STAGE_STATUS_<stage>.md`。交接文档是快照，不得与状态文件形成两套冲突状态。

## 5. 用户操作队列与批量测试包机制

### 5.1 机制目标

通过提前拆分并行任务，把助手无法直接完成或只会产生等待的工作交给用户执行，从而减少：

- 长编译占用对话周期；
- 实机测试逐条命令往返；
- 开发完成后才发现环境信息缺失；
- 多次重复采集同一主机状态；
- 用户等待期间没有可推进任务。

该机制不把设计责任转移给用户。用户是操作员、现场观察者和决策人；助手仍对架构、源码、测试设计、安全边界、命令正确性和结果分析负责。

### 5.2 双队列

每个活动阶段必须维护：

#### 助手开发队列

```text
A1 源码与架构实现
A2 host 模型与故障矩阵
A3 单元测试与 compile-only 定义
A4 CI、manifest 和证据格式
A5 日志分析、结论与交接
```

#### 用户操作队列

```text
U1 只读环境快照
U2 长时间编译或本地测试
U3 物理硬件检查与测量
U4 长周期稳定性观测
U5 经授权的实机或主机操作
U6 产品、风险与发布决策
```

每个用户任务必须标记：

```text
TASK_ID=
STATUS=ready|wait_authorization|blocked|running|done|failed|cancelled
RISK_CLASS=A|B|C|D
CAN_RUN_IN_PARALLEL=true|false
DEPENDS_ON=
ESTIMATED_DURATION=
AUTHORIZATION_REQUIRED=true|false
EXPECTED_RETURN=
```

### 5.3 适合交给用户的工作

优先交付：

- ESPHome、ESP-IDF、Docker 和完整 pytest 的长编译/长测试；
- OS、内核、Docker、容器、端口、网络、磁盘和版本只读快照；
- ESP32-C6、M401A、T1 等实体设备的接线、供电、串口、LCD、LED、传感器和电压观察；
- 断电、断网、重启、恢复供电等已授权故障注入；
- 数小时或数天的稳定性、重连、成功率、资源占用和电池变化观测；
- 保存完整终端输出、照片、CI 链接和测试日志；
- 产品边界、风险豁免和发布决策。

### 5.4 不应转移给用户的工作

除非用户明确要求参与编码，否则不得要求用户承担：

- 协议、状态机、身份、ACL、授权和回滚语义设计；
- 故障归因和日志分析；
- 候选身份、manifest 和证据 schema 设计；
- 自行拼接危险命令；
- 自行判断是否可绕过门禁；
- 在缺乏候选哈希、授权和清理路径时“先试一下”。

### 5.5 并行拆分规则

S0/S1 阶段必须主动识别可并行任务：

- 不依赖待开发源码的只读采集应立即进入 `ready`；
- 依赖已有提交但耗时较长的 compile-only 可与后续文档/模型开发并行；
- 需要新候选或授权的实机任务进入 `wait_authorization`，不得提前执行；
- 同一设备上可能互相干扰的测试不得并行；
- 生产环境、真实 Broker、物理 NVS、烧录和 eFuse 默认不进入 `ready`；
- 若用户任务结果会改变助手当前实现方向，应先等结果；否则助手继续推进到下一决策门。

### 5.6 禁止碎片化交互

默认禁止：

> 给一条命令 → 等用户返回一小段输出 → 再给下一条命令。

除非前一步结果决定下一步是否安全，否则应一次交付完整批量测试包。

同一目标的只读命令、校验命令、执行命令、证据收集和清理命令应组合为一个可审查的操作包。需要分段时，按明确门划分，例如：

1. 只读预检包；
2. 候选上传与哈希校验包；
3. 经授权执行包；
4. 复核与清理包。

不得把可一次完成的十个机械步骤拆成十轮对话。

## 6. 批量测试包规范

### 6.1 必备内容

每个交给用户执行的测试包必须包含：

1. **任务目的**：要证明什么，不能证明什么；
2. **风险等级**：只读、隔离写、实机可逆写或不可逆；
3. **目标身份**：设备、主机、仓库、stage、source SHA、candidate generation；
4. **前置条件**：目录、工具、网络、供电和授权状态；
5. **完整命令**：不得使用省略号或缺失参数；
6. **文件校验**：上传前后 SHA-256；
7. **成功标志**：退出码和关键输出；
8. **停止条件**：出现什么现象立即停止；
9. **回滚/清理**：无论成功失败都能执行；
10. **证据要求**：用户需要返回哪些日志、照片、摘要或文件；
11. **隐私与脱敏**：不得回传密码、令牌、私钥和完整敏感路径；
12. **预计耗时**：便于用户安排并行执行。

### 6.2 脚本安全基线

用于实机或主机操作的 shell 脚本原则上必须包含：

```bash
set -euo pipefail
umask 077
```

并根据任务实现：

- `trap` 清理；
- 前后状态快照；
- 受保护对象 SHA-256 或等价指纹对比；
- 默认 dry-run 或 `disabled_by_default`；
- 明确的授权参数；
- 非零失败退出码；
- 失败阶段和错误码；
- 不在输出中泄露密码、完整凭据或隐私标识；
- 不要求在 T1/M401A 临时安装新依赖，优先使用现有标准工具。

### 6.3 标准交付格式

```text
BATCH_PACKAGE_ID=
TASK_IDS=
STAGE=
SOURCE_SHA=
CANDIDATE_GENERATION=
TARGET=
RISK_CLASS=
AUTHORIZATION_ID=
AUTHORIZATION_EXPIRES=
ESTIMATED_DURATION=

PURPOSE:

PRECONDITIONS:

FILES:
- LOCAL_PATH=
- REMOTE_PATH=
- SHA256=

COMMANDS:
1. preflight
2. upload
3. verify
4. execute
5. collect evidence
6. cleanup

EXPECTED_SUCCESS:

STOP_IMMEDIATELY_IF:

RETURN_TO_ASSISTANT:
```

### 6.4 用户回传证据格式

优先要求用户返回一个完整证据包，而不是零散截图：

```text
BATCH_PACKAGE_ID=
START_TIME=
END_TIME=
HOST_OR_DEVICE=
BASE_SHA=
SOURCE_SHA=
SCRIPT_SHA256=
COMMAND_GROUP=
EXIT_CODE=
BEFORE_STATE=
AFTER_STATE=
CLEANUP_RESULT=
ARTIFACT_PATH_OR_LINK=
USER_OBSERVATION=
SECRET_VALUES_INCLUDED=false
```

大量输出应保存为文件或 Artifact；对话中只粘贴机器可读摘要、错误附近日志和必要观察结果。

### 6.5 长周期任务

长周期测试必须明确：

- 采样周期和总时长；
- 日志轮转和最大空间；
- 中途失败是否停止；
- 断电/断网是否属于测试内容；
- 通过阈值；
- 结束后自动清理；
- 用户无需持续盯守时，应明确说明可无人值守运行；
- 不允许助手声称在对话外后台继续执行。

---

## 7. 分支、PR、提交与脚本规范

### 7.1 分支

推荐：

```text
feature/<system>-<stage>-<purpose>-<date>
fix/<system>-<stage>-<purpose>-<date>
docs/<purpose>-v<version>-<date>
```

分支必须唯一、可读、包含阶段或目的；不得复用历史冻结分支或在无关分支顺带开发新功能。

### 7.2 Draft PR 作为阶段控制面

PR 应包含目标、不做事项、基线 SHA、gate、风险、候选 generation、实机状态、Artifact、用户决策项，以及助手/用户双队列摘要。

关键状态必须同步回 `STAGE_STATUS` 或 manifest，不能只存在于 PR 评论。

### 7.3 脚本框架

优先采用：

```text
tools/framework/
  authorization.py
  candidate.py
  evidence.py
  rollback.py
  remote_exec.py

tools/entrypoints/
  <stage-unique-entry>.py

tests/
  fixtures/
  matrices/
```

稳定框架可复用；阶段入口、授权请求和实机执行文件保持唯一命名。唯一命名用于防误执行，不等于复制整套实现。

新建脚本前必须确认是否已有同类功能、能否参数化、是否只是更换常量/路径/阶段名、是否应改成数据驱动，以及是否会引入另一套证据 schema。

## 8. GitHub CI、证据与候选成本控制

### 8.1 CI

必须逐步落实：path filter、concurrency、cancel-in-progress、缓存、快/重门分离、候选显式触发、Artifact 保留期、失败 job 单独重跑，以及 compile-only 与真实部署分离。

推送节奏通常为：骨架、CI-ready、CI 修正、候选冻结、实机证据/文档收尾。不是硬限制，但明显偏离时应检查碎片化。

禁止以 CI 代替本地基本测试、每次小编辑触发全量构建、为偶发失败提交无意义变更、把大日志提交到 Git 或维护含义相同的重复 workflow。

### 8.2 证据分层

- **L1 摘要证据**：结论、状态、关键哈希，进入 Git；
- **L2 结构化证据**：完整 JSON、矩阵结果，进入 Artifact，Git 保存索引和哈希；
- **L3 原始证据**：完整日志、镜像、抓包、长周期采样，进入受控归档。

证据绑定 repository、source SHA、stage、generation、工具版本、目标身份、执行时间、授权身份和 artifact SHA256。

结论状态只能使用：`passed`、`failed`、`not_run`、`inconclusive`、`waived`。不得把未运行或证据不足写成“基本通过”。

## 9. 决策与中断规则

除非用户另有要求，只在以下节点中断：

1. **D1 架构/范围决策**；
2. **D2 实机/生产授权**；
3. **D3 风险豁免**；
4. **D4 Ready、合并、发布或不可逆动作决策**。

普通代码结构、fixture、文档排版、不改变产品行为的重构、本地模型实现、常规 CI 修复不应频繁中断。

用户要求持续推进时，助手应推进到下一个真实决策门；同时提前交付可并行的用户操作包，而不是让用户无任务等待。

## 10. 阶段效率指标

每阶段结束记录：

```text
PUSH_COUNT=
CI_RUN_COUNT=
CI_FAILED_RUN_COUNT=
FAILURES_FOUND_LOCALLY=
FAILURES_FIRST_FOUND_IN_CI=
CANDIDATE_GENERATION_COUNT=
BRANCH_COUNT=
NEW_SCRIPT_COUNT=
REUSED_FRAMEWORK_COUNT=
REAL_MACHINE_RUN_COUNT=
ROLLBACK_COUNT=
USER_TASK_COUNT=
USER_TASKS_COMPLETED=
BATCH_PACKAGE_COUNT=
PARALLEL_USER_TASK_COUNT=
SERIAL_COMMAND_ROUND_COUNT=
USER_WAIT_ONLY_ROUND_COUNT=
USER_RETURN_REWORK_COUNT=
```

重点观察：

- CI 首次发现的低级错误是否下降；
- 每阶段分支是否接近 1；
- 新增完整脚本数量是否下降；
- 候选是否因过早冻结而过多；
- 实机是否只验证模型无法证明的事项；
- 用户等待期间是否有安全、有效的并行任务；
- 同一测试是否仍被拆成大量命令往返；
- 用户回传证据是否一次满足分析需要。

## 11. 渐进式落地计划

### 立即执行

- 新阶段采用一阶段一主分支；
- 建立单一 `STAGE_STATUS`；
- 推送前运行本地 preflight；
- PR 默认 Draft；
- 大日志保存为 Artifact；
- 候选延后至 CI 通过；
- 每阶段建立助手/用户双队列；
- 用户操作默认通过批量测试包交付。

### 未来 1—2 个阶段

- 收敛统一 preflight；
- 抽取授权、证据、回滚和远程执行框架；
- Actions concurrency 和 path filter；
- 故障矩阵数据驱动；
- 候选 manifest schema V1；
- 批量测试包和用户证据包 schema；
- 自动生成用户操作队列摘要。

### 流程稳定后

- 候选与授权自动绑定；
- 自动生成阶段状态、交接提示词和操作包索引；
- M401A/T1/节点固件形成统一但分目标的验收框架；
- 定期清理失效 Artifact，保留长期发布证据；
- 用效率指标评估门禁价值，连续两个阶段无有效发现时通过 ADR 调整，不得临时绕过。

不追溯重写已完成历史阶段，不修改冻结分支和证据；从下一个新阶段开始完整执行。

## 13. 成功标准

实施后应出现：

- GitHub CI 主要承担独立复核，而不是基础调试；
- 同阶段分支、近似脚本和候选 generation 明显减少；
- 实机测试次数减少但证据质量提高；
- 用户等待期间能够并行完成安全、明确、有价值的任务；
- 大多数用户操作由一个批量测试包完成，而不是多轮零散命令；
- 用户不需要承担架构设计和故障归因；
- 生产安全边界保持不变；
- 新对话能通过状态文件、双队列和提示词快速恢复；
- 项目时间更多用于产品功能、故障模型和用户价值，而不是重复整理 GitHub 状态和等待机械执行。

本规范是可演进流程文件。任何优化都不得以减少安全证据、模糊失败结论或扩大未经授权的实机操作为代价。
